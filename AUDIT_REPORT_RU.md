# Аудит Demerzel v7

## Что представляет собой проект

Проект объединяет в одном процессе:

- Telegram-бота на aiogram;
- веб-панель на Flask + Waitress;
- SQLite-базу;
- фоновые циклы резервного копирования, мудростей, отложенных сообщений и донат-уведомлений;
- файловое хранилище изображений.

Точка запуска — `demerzel.py`. Веб работает в отдельном потоке, бот — в основном asyncio event loop.

## Главные найденные проблемы

1. **Не было общей инициализации базы.** Часть таблиц создавалась только при открытии отдельных страниц или запуске отдельных функций. На пустом volume Bothost отсутствовали `chat_users`, `other`, `scheduled_campaigns`, `form_text`, `levels` и другие таблицы.
2. **В архиве находились две разные базы:** `database/database.db` и `data/database.db`. В них отличались таблицы и данные. При смене `DATA_DIR` приложение работало с разными схемами.
3. **Старая база не мигрировалась полностью.** `CREATE TABLE IF NOT EXISTS` не добавляет отсутствующие столбцы.
4. **Старая версия `permission_messages` имела другую структуру** без `user_id`.
5. **Страница участников отсутствовала полностью.** В архиве не было `web/members.py` и шаблона.
6. **Команда `/bv` снова была только для администраторов** и не удаляла предыдущий ответ.
7. **Веб-просмотр изображений удалял исходные PNG/GIF**, после чего Telegram-бот пытался открыть уже отсутствующий файл.
8. **Dockerfile был чрезмерно сложным**, использовал Python 3.13 и размещал код рядом с постоянными данными.
9. **Несколько экземпляров polling могли использовать один токен**, вызывая `TelegramConflictError`.
10. **`/health` проверял только HTTP-процесс**, но не состояние базы.

## Что исправлено

- Добавлен `bot/schema.py` — единая безопасная инициализация и миграция SQLite.
- База готовится до первого подключения веба или бота.
- Добавляются отсутствующие таблицы и столбцы без удаления существующих данных.
- Старая таблица `permission_messages` перестраивается с сохранением записей.
- Начальные настройки копируются только в пустые конфигурационные таблицы; данные пользователей не подменяются.
- Добавлены индексы для участников и планировщика.
- Добавлена страница `/members`:
  - список участников;
  - поиск;
  - CSV UTF-8/Windows-1251;
  - импорт в `admin_members`;
  - необязательная синхронизация с `chat_users`;
  - выдача доната по типу, сроку и суточному лимиту.
- `/bv` доступна любому немьютнутому участнику.
- Для одного адресата сохраняется только последнее сообщение `/bv`.
- Одновременные `/bv` для одного адресата защищены asyncio-lock.
- Веб больше не удаляет исходники изображений.
- Бот умеет использовать созданную JPG-копию, если исходник уже отсутствовал.
- Добавлена файловая блокировка polling внутри общего persistent volume.
- Docker переведён на Python 3.12.
- Код расположен в `/usr/src/app`, постоянная база — в `/app/data`.
- `/health` проверяет наличие ключевых таблиц и возвращает JSON.
- Добавлена понятная страница 500, traceback остаётся в журнале приложения.

## Команды Telegram-бота

Обнаружены и импортированы команды:

- Пользовательские: `/bv`, `/top10`, `/stats`, `/view`.
- Работа с анкетами: `/save`, `/canview`.
- Модерация: `/ban`, `/kick`, `/mute`, `/unmute`, `/lock`, `/unlock`.
- Донаты и доступы: `/media`, `/voice`, `/emoji`, `/circle`, `/tag`, `/viewd`, `/viewmd`.
- Снятие доступов: `/delem`, `/delgs`, `/delcircle`, `/deltag`, `/delm`, `/delm2`, `/delmax`.
- Служебные: `/st`, `/donattest`.

Проверены импорты обработчиков, регистрация роутеров, используемые update-типы и SQL-зависимости. Полный сетевой тест команд требует тестового Telegram-чата и действительного токена; в архиве токены не сохранены.

## Ограничения проверки

- В контейнере проверки не установлен Docker CLI, поэтому образ не собирался командой `docker build`; сам Python-проект и зависимости запускались в отдельном виртуальном окружении.
- Telegram API не вызывался с реальным токеном, чтобы не вмешиваться в работающего бота и не создавать второй polling.

## Развёртывание на Bothost

1. Остановить старую копию проекта и все другие процессы с тем же `BOT_TOKEN`.
2. Сделать резервную копию текущего `/app/data/database.db`.
3. Загрузить архив и выполнить полную пересборку Docker-образа.
4. Оставить постоянный volume на `/app/data`.
5. Передать переменные:
   - `BOT_TOKEN`;
   - `FLASK_SECRET_KEY`;
   - `ADMIN_USERNAME`;
   - `ADMIN_PASSWORD`;
   - `BACKUP_CHANNEL_ID`;
   - `LOG_CHANNEL_ID`;
   - `SOURCE_CHAT_ID`;
   - `COSMOS_ID`;
   - `PORT` — порт Bothost.
6. Проверить `/health`. Ожидаемый ответ:

```json
{"status":"ok","missing_tables":[]}
```

7. В журнале должна появиться строка `Database ready: path=/app/data/database.db`.
